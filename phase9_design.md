# Phase 9 — DESIGN

> The audit suite, the release pipeline, the launch checklist, and
> ADRs 046–048. Implements the principles in `phase9_think.md` as
> concrete tests, scripts, docs, and gates.

The deliverables here are *runnable* — the synthetic-cohort harness
is a real pytest module, the load tests are k6 scripts the operator
runs against a staging URL, and the launch readiness report is
generated by a script that walks the gate matrix and signs the
output.

---

The flow is two-pass. The first pass *measures* (pedagogy, load,
κ, security). The second pass *signs and tags*. The signature step
refuses to proceed if any measurement is below the gate.

## 2. The pedagogy audit

### 2.1 The scenario suite

`tests/pedagogy/launch_audit.py` defines twelve launch cohorts. The
file uses Phase 4's `synthetic_cohorts.COHORTS` as its baseline
fixtures and extends them with launch-specific scenarios that
exercise the *time* axis (84-day vs 42-day vs daily-budget
shortfalls), not just the *learner archetype* axis (already
covered by Phase 4).

| # | id                              | initial NCLC                | target | days | daily_min | kind            |
|---|---------------------------------|-----------------------------|--------|------|-----------|-----------------|
| 1 | solid_B1_target_NCLC7           | flat NCLC 5                 | 7      | 84   | 150       | realistic       |
| 2 | solid_B1_target_NCLC9           | flat NCLC 5                 | 9      | 84   | 150       | honest_refusal  |
| 3 | uneven_target_NCLC7             | CO/CE 6, EE/EO 4            | 7      | 84   | 150       | realistic       |
| 4 | strong_B2_target_NCLC9          | flat NCLC 7                 | 9      | 84   | 150       | realistic       |
| 5 | aggressive_B1_target_C2         | flat NCLC 5                 | 11     | 84   | 150       | honest_refusal  |
| 6 | short_runway_target_NCLC7       | flat NCLC 5                 | 7      | 42   | 150       | honest_refusal  |
| 7 | low_budget_target_NCLC7         | flat NCLC 5                 | 7      | 84   | 60        | honest_refusal  |
| 8 | already_at_target               | flat NCLC 9                 | 9      | 84   | 150       | trivial         |
| 9 | heritage_production_gap         | CO/CE 8, EE/EO 4            | 7      | 84   | 150       | realistic       |
| 10| reception_only_weakness         | CO/CE 4, EE/EO 7            | 7      | 84   | 150       | realistic       |
| 11| ee_bottleneck_only              | CO/CE 7, EE 4, EO 7         | 8      | 84   | 150       | realistic       |
| 12| eo_bottleneck_only              | CO/CE 7, EE 7, EO 4         | 8      | 84   | 150       | realistic       |

The "honest_refusal" classification means: under the planner's
conservative learning-rate constant (0.05 NCLC/hour with
diminishing returns), this cohort *cannot* reach its target.
The audit verifies the planner refuses (projected_min < target);
it does not require the system to perform a miracle.

For each cohort the harness:

1. Builds the cohort's confident-posterior fixture
   (`Cohort.confident_posteriors()` from Phase 4).
2. Calls the planner (`generate_plan(...)`) to obtain the daily
   allocation trajectory.
3. Runs `simulate_learning(...)` for `days` iterations under
   the per-day allocation.
4. Repeats with stochastic noise (Gaussian, σ = 25% of daily
   gain) for N = 100 trajectories.
5. Reports:
   - `P(min_skill ≥ target_NCLC at day = days)` — the headline.
   - Per-skill final-day posterior mean + 5–95 percentile band.
   - The planner's *projected* min-skill at horizon vs the
     median simulated min-skill (the calibration delta).
6. Writes the per-cohort + aggregate JSON to
   `data/audit/phase9/pedagogy_audit.json` for the signer.

### 2.2 Launch gate

The audit applies three gates per cohort.

**Gate 1 — calibration (universal).** The planner's analytic
projection of the min-skill must agree with the simulator's
empirical median within ±0.5 NCLC (`CALIBRATION_WINDOW_NCLC`).
This is the load-bearing test: it verifies the planner's number
is a *real* projection, not marketing. The 0.5-NCLC window is
tight because the planner uses the same conservative learning-
rate constant the simulator uses.

**Gate 2 — projection-outcome consistency (universal).** If the
planner projects success, simulated `P(success) ≥ 0.65`. If the
planner refuses (projected_min < target), simulated `P(success)
≤ 0.10`. A mismatch in either direction means the planner is
lying.

**Gate 3 — per-cohort-class assertion.**

- *Realistic* cohorts: planner projects success AND simulated
  `P(success) ≥ 0.65`.
- *Honest-refusal* cohorts: planner refuses to project success
  (projected_min < target). This includes the obviously-over-
  reach cases (`aggressive_B1_target_C2`) and the cohorts at the
  upper edge of the documented target band where the conservative
  rate-constant + diminishing-returns curve correctly project
  insufficient gain (`solid_B1_target_NCLC9` from flat NCLC 5;
  `short_runway_target_NCLC7` at 42 days; `low_budget_target_NCLC7`
  at 60 min/day). The audit verifies the planner *refuses*; it
  does not require the planner to succeed at a stretch goal.
- *Trivial* cohorts (already at target): no regression
  (`P(success) ≥ 0.95`).

### 2.3 Output

- `tests/pedagogy/launch_audit.py` is a pytest module: `pytest -q
  tests/pedagogy/launch_audit.py` returns non-zero on gate
  failure.
- `pytest --launch-audit-report=data/audit/phase9/pedagogy_audit.json`
  (custom `pytest_addoption`) writes the structured JSON.
- A Mermaid calibration plot is emitted as
  `data/audit/phase9/pedagogy_calibration.md` (per-cohort scatter,
  planner projection on x-axis, simulated median on y-axis, ±10pp
  band shaded). Mermaid is used so the doc lives in the repo
  without binary assets.

### 2.4 Calibration of the calibration

The harness uses the *same* `LEARNING_RATE_PER_MINUTE` constant
the planner uses. That is deliberate — we are auditing that the
planner's projection (which compounds the constant 84 times under
diminishing returns) actually matches what 84 noisy applications
of the same constant produce. If the gate fires, the bug is in the
planner's analytic projection (or the diminishing-returns curve),
*not* in a mismatched simulator constant. This makes the audit
honest.

## 3. The security audit

### 3.1 OWASP Top 10 — verified controls

| OWASP | Control | Evidence |
|---|---|---|
| A01 Broken Access Control | Per-user-scope on every `/v1/.../{id}` route; second-user fuzz | `tests/integration/test_authz_fuzz.py` |
| A02 Cryptographic Failures | TLS at the proxy (operator responsibility, doc'd); JWT HS256 with rotating secret | `OPERATIONS.md §4` |
| A03 Injection | Pydantic validation at every boundary; SQLAlchemy parameterized queries | `tests/lint/test_no_raw_sql.py` |
| A04 Insecure Design | Phase 1–8 design docs; ADRs trace every load-bearing choice | `docs/adrs/*` |
| A05 Security Misconfiguration | Headers set in `apps/api/.../middleware/security_headers.py`; `securityheaders.com` clean | `data/audit/phase9/security_headers.json` |
| A06 Vulnerable Components | `pip-audit`, `npm audit`, `trivy fs` clean in CI | `.github/workflows/ci.yml` |
| A07 Identification & Authn | Short-lived access (15 min) + refresh (7 d) with rotation | `apps/api/src/tcf_accel_api/routes/auth.py` |
| A08 Software & Data Integrity | Pinned deps; SBOM in release; checksums signed | `scripts/release/build_release.py` |
| A09 Security Logging | Structured logs; PII scrubber; no learner text/audio in logs | `tests/unit/test_logging_no_pii.py` |
| A10 SSRF | LLM gateway URL allow-list; no user-controllable URL params | `packages/ml/.../litellm_gateway.py` |

### 3.2 The toolchain

- **Static**: `bandit -r packages/ apps/` + `ruff` + `mypy --strict`.
- **Dependencies**: `pip-audit -r apps/api/pyproject.toml`,
  `pnpm audit --prod`, `trivy fs .`.
- **Secrets**: `gitleaks detect --redact` in pre-commit and CI.
- **Container scan**: `trivy image ghcr.io/<org>/tcf-accel-api:v1.0.0`.
- **Dynamic (optional, doc'd)**: ZAP baseline against staging:
  `docker run owasp/zap2docker-stable zap-baseline.py -t https://staging.example`.
- **Authz fuzz**: `tests/integration/test_authz_fuzz.py` enumerates
  every `/v1/.../{id}` route, generates two test users, and
  verifies user-B cannot read/write user-A's resources.

### 3.3 Logging hygiene

`apps/api/src/tcf_accel_api/logging.py` (Phase 9 add) registers a
`PIIScrubFilter` that:

- Drops any field named `text`, `transcript`, `audio_path`,
  `email`, `phone`, `name`.
- Truncates any string > 200 chars (audit-evident truncation, not
  silent loss).
- Adds a fingerprint hash for cross-event correlation without
  retaining the underlying content.

Tested by `tests/unit/test_logging_no_pii.py`: feeds a mock
submission through the API, captures the log emitter, and asserts
no candidate text appears in the captured records.

### 3.4 Headers

`apps/api/src/tcf_accel_api/middleware/security_headers.py` (Phase
9 add) sets, on every response:

- `Strict-Transport-Security: max-age=31536000; includeSubDomains; preload`
- `Content-Security-Policy: default-src 'self'; img-src 'self' data:; script-src 'self'; style-src 'self' 'unsafe-inline'` (Next strict-dynamic variant in `apps/web/next.config.mjs`)
- `X-Frame-Options: DENY`
- `X-Content-Type-Options: nosniff`
- `Referrer-Policy: strict-origin-when-cross-origin`
- `Permissions-Policy: microphone=(self), camera=(), geolocation=()`

Note the `microphone=(self)` — the EO drills require it. Camera +
geolocation are denied; we do not use them.

Verified by `tests/integration/test_security_headers.py` and an
optional `securityheaders.com` run against staging (doc'd in
`OPERATIONS.md §6`).

## 4. Performance audit

### 4.1 The k6 suite

`tests/load/k6_sustained.js` — 100 VUs for 10 min, mixed drill
workload:

- 60% `GET /v1/plan/today`
- 20% `POST /v1/session/answer`
- 10% `GET /v1/insights/skill_summary`
- 10% `GET /v1/mock_exam/state`

Thresholds:

```js
thresholds: {
  http_req_duration: ['p(95)<250'],
  http_req_failed:   ['rate<0.001'],
}
```

`tests/load/k6_burst.js` — 500 VUs ramp 0→500 over 30s, hold 30
min, all hitting `POST /v1/mock_exam/start`. Thresholds:

```js
thresholds: {
  'mock_exam_started': ['count>14500'],   // ~all starts succeed
  'mock_exam_score_failed': ['count<50'], // < 0.1% scoring failures
}
```

`tests/load/k6_cold_start.js` — boots `docker compose up`, polls
`/healthz` on api + worker + db, asserts all three green within
60s.

### 4.2 Capacity targets

- API p95 latency ≤ 250 ms at 100 concurrent learners.
- Burst: 500 concurrent mock-exam starts, error rate < 0.1%, no
  scoring jobs dropped over 30 min.
- Cold start: 60 s.
- Disk: full seeded bank ≤ 20 GB. Verified by
  `scripts/seed_bank.py --report-disk` post-seed.

Reports land in `data/audit/phase9/perf_*.json`.

### 4.3 What we don't measure here

- Frontend LCP/INP at the *deployed* URL — Lighthouse is run as
  part of `web-ci` (Phase 8 §17) and re-run against staging.
  Phase 9 takes the Lighthouse report from `apps/web/lhci/` and
  attaches it; no separate measurement.
- Database read/write IOPS — out of scope for v1 (single-tenant);
  documented as a v1.1 concern in `OPERATIONS.md §11`.

## 5. Content audit

### 5.1 Distribution check

Re-runs `scripts/seed_bank.py --audit-only` which:

- Reads the seeded bank from Postgres.
- Cross-references against the quota matrix in
  `packages/content/.../quota_matrix.py`.
- Reports per-module counts vs target counts, per-CEFR-band
  histograms, and synthetic-vs-scraped ratios (per ADR-021).

Pass criteria: every cell within ±5% of target; synthetic ratio ≤
40% per module.

### 5.2 Hand review

100-item-per-module sample (400 items total). Reviewer (one per
module, must not be the author) verifies:

- CEFR label matches the item's complexity (judgment).
- MCQ key is correct.
- For CO: audio plays, transcript matches (post-REVEAL phase),
  speaker is intelligible.
- For EE/EO prompts: rubric is appropriate, task is realistic.

Output: `data/audit/phase9/content_handreview.md` — per-item row
with reviewer initials, agree/disagree, notes. Pass: ≥ 95% agree
per module.

### 5.3 FEI drift check

Compares 10 of our CO items per CEFR band against the 2025/2026
published FEI samples (linked from the master prompt §1.1).
Flags any items where our difficulty, register, or genre
deviates noticeably. Output: `data/audit/phase9/fei_drift_note.md`.

This is qualitative; the criterion is "no item is grossly
mis-leveled or off-genre," not a numeric κ.

### 5.4 PII + copyright + harm

- Grep the seeded bank for likely PII (regex: emails, phone
  numbers, postal codes); zero hits required.
- Grep for known copyrighted strings (a small allow-list of
  watermark phrases from RFI / TV5MONDE / Radio-Canada);
  zero hits required.
- Topic filter: items must not reference self-harm, hate speech,
  or politically inflammatory content in a way that exceeds
  TCF-Canada-style register. Manual spot check during the
  hand-review.

The fallback is the realistic v1.0 path; the external auditor is

### 6.3 Deaf-friendly path documentation

`LEARNER_GUIDE.md §8` describes how a Deaf or hard-of-hearing
candidate uses the system in CE/EE/EO-only mode:

- Diagnostic skips CO; the planner's allocation re-weights to the
  three remaining skills (the allocator's behavior is the same as
  the "CO already at target" case).
- Mock exams skip CO; the readiness widget refuses 🟢 until two
  consecutive CE/EE/EO mocks land green.
- The Library page links to IRCC's accommodations page and to
  Canadian deaf-advocacy organizations.

The structural exam constraint — that CO is audio-only by FEI
design — is documented in `LIMITATIONS.md §3` and surfaced in
the diagnostic UI.

## 8. Launch checklist (must be 100% green)

This is the *checklist the signer iterates*:

- [ ] All previous phase gates passed; `phaseN_evaluate.md`
      shows ✅ or ⚠-with-handoff-to-Phase-9.
- [ ] Pedagogy audit pass (§2 gate).
- [ ] Security audit pass (§3 controls + clean toolchain).
- [ ] Performance audit pass (§4 thresholds).
- [ ] Content audit pass (§5: distribution + hand-review + FEI
      drift + PII/copyright/harm).
- [ ] Accessibility audit pass (§6: Lighthouse ≥ 90 + pa11y
      clean + WCAG 2.2 AA sign-off).
- [ ] Documentation audit pass (§7: each doc edited by a second
      maintainer; no TODO markers).
- [ ] Risk register: every Open risk has a Mitigated, Accepted,
      or Retired disposition. Zero Open.
- [ ] FEI source-of-truth check: re-verify §1.1 of master prompt
      against current FEI page; record date + auditor in the
      readiness report.
- [ ] `v1.0.0` tag created; Docker images published; Helm chart
      published; PyPI shared+sla packages published; SBOM
      attached.
- [ ] Demo deployment running and observed for 48 h with no P0
      alerts.
- [ ] Signed Launch Readiness Report (one page; auditor name +
      date + SHA-256 of the audit bundle).

`scripts/release/sign_audit_report.py` walks this list, reads
each piece of evidence from `data/audit/phase9/`, and fails the
release on any unchecked item.

## 10. ADRs

### ADR-046: Launch criteria are blocking, not advisory

**Status:** Proposed in Phase 9; accepted at release.

**Context.** The launch checklist (§8) is the gate between "done
with the build" and "tagged v1.0.0." Without a binding rule, a
maintainer could rationalize shipping with a yellow gate; six
months in, "yellow" becomes the cultural default and the launch
discipline erodes.

**Decision.** Every item on the launch checklist is *blocking*. A
red checkbox blocks the tag. A yellow checkbox requires an
explicit Accepted-risk entry in `RISK_REGISTER.md` and a one-
paragraph rationale signed by two maintainers; without it, it is
red. There is no "we'll fix it in 1.0.1" path: if it isn't in 1.0,
it isn't in 1.0.

**Consequences.** Slower path to the tag; honest tag when we get
there.

### ADR-047: External security review for v1.0 (or two-maintainer sign-off as a fallback)

**Status:** Proposed in Phase 9; accepted at release.

**Context.** Phase 9 §3 names the controls and the toolchain.
That covers static, dependency, secrets, and authz-fuzz testing.
It does not cover an *adversarial* review by someone who didn't
write the code. An external pentest of an OSS project is rarely
budgeted; we need a clear policy.

**Decision.** v1.0 ships with one of:
1. an external WCAG + security audit signed off by a third party
   (preferred); OR
2. two-maintainer independent review documented in
   `data/audit/phase9/sec_review_internal.md`, each maintainer
   running the toolchain on a clean checkout and the second one
   spot-checking the authz fuzzer's coverage.

v1.1 commits to acquiring (1).

**Consequences.** v1.0 ships with the fallback. We are honest
about the limitation in the launch post.

### ADR-048: Public publication of κ + cohort-success simulation results in the README

**Status:** Proposed in Phase 9; accepted at release.

**Context.** ADR-038 (Phase 7) made the published κ a release
gate but left *where* it is published unspecified. The
pedagogy-audit cohort results (Phase 9 §2) are also a credibility
signal a reader should be able to find without digging into the
audit dossier.

**Decision.** Both the κ table and the cohort P(success) table
land in the README under a `## Honesty receipts` section,
linked from the badge row. The README is regenerated from
`data/audit/phase9/pedagogy_audit.json` and
`data/calibration/ee.v1.report.md` (and the equivalent EO file
when shipped) by `scripts/release/sign_audit_report.py`'s
`--update-readme-receipts` flag.

**Consequences.** The README grows by ~30 lines. The numbers
visible at the top of the repo match what the audit bundle says.
A reader can verify our honesty in 30 seconds without cloning.

## 11. The release pipeline (script-level)

`scripts/release/build_release.py`:

1. Validate working tree clean; current branch is `release/v1.0.0`
   or `main` with an annotated tag pending.
2. Run `make verify` (lint + typecheck + unit + integration).
3. Run `pytest tests/pedagogy/launch_audit.py` (the gate).
4. Run `python scripts/eval_kappa.py --release v1.0.0` (gate
   against Phase 7 κ baseline).
5. Build wheels for `tcf-accel-shared`, `tcf-accel-sla`,
   `tcf-accel-ml`, `tcf-accel-content`.
6. Build Docker images for `api`, `worker`, `web` (multi-arch:
   linux/amd64, linux/arm64).
7. Render Helm chart values from `infra/helm/values.yaml`.
8. Generate SBOM via `syft .` → `data/audit/phase9/sbom.spdx.json`.
9. Generate SHA-256 manifest covering all built artefacts +
   audit JSON files.
10. (If `--sign`) `cosign sign-blob` over the manifest.

`scripts/release/sign_audit_report.py`:

1. Read the launch checklist YAML
   (`scripts/release/launch_checklist.yaml`).
2. For each item, read the named evidence file from
   `data/audit/phase9/`; verify it exists and (where applicable)
   parse it for the documented pass marker.
3. Emit `LAUNCH_READINESS_REPORT.md` — one page, gate table,
   auditor name + date, SHA-256 of the audit bundle.
4. Exit non-zero if any gate item fails.

A sample `LAUNCH_READINESS_REPORT.md` is rendered into the repo
even when some gates are not yet measured; the script marks them
as `PENDING-Phase 9` and the eval step measures them.

## 12. What Phase 9 does *not* code

We deliberately don't write new product code in Phase 9. The job
is to *verify* the product, not to extend it. The only "new code"
is:

- The audit harness (`tests/pedagogy/launch_audit.py`).
- The load tests (`tests/load/k6_*.js`).
- The release scripts (`scripts/release/*.py`).
- The security middleware files (`security_headers.py`,
  `pii_scrubber.py`) — small, single-purpose, defensible.

Any product bug surfaced by the audit gates is fixed in-place
with a minimal patch; we don't sneak features in under the launch
banner.
